# 审批：功能设计

> 实现标记：本文描述目标设计；当前 `0.1.0` 已实现项和后续边界以 [后端实现状态](../../implementation-status.md) 为准。

## 1. 目标

让用户在本地或手机端安全地理解并处理 Codex App Server 发出的审批请求，同时避免重复批准、错误批准或对已经失效的请求进行操作。

审批是高优先级功能，其实时性和正确性高于普通文本 delta。

## 2. 审批展示

审批卡片至少展示：

- 目标设备、项目、Thread 和 Turn。
- 审批类型。
- App Server 提供的操作摘要。
- 工作目录或影响范围提示。
- 发起时间和剩余有效状态。
- App Server 实际支持的决策选项。
- 安全警告和数据是否经过截断。

不能为了简洁隐藏会影响判断的命令、目标范围或权限变化。

## 3. 审批类型

Nuntius 使用规范化分类：

- 命令执行审批。
- 文件修改审批。
- 网络或外部工具审批。
- MCP/工具交互审批。
- 未识别审批。

实际决策项以 App Server 请求为准，Nuntius 不自行发明“永久允许”等 App Server 未提供的选项。

## 4. 状态

```text
pending -> responding -> approved
                      -> denied
pending -> expired
pending -> cancelled      Turn 已结束或中断
任意不确定 -> unknown
```

- 只有 pending 可以提交决策。
- responding 期间按钮禁用，但页面可查询状态。
- approved/denied 只有设备或 App Server 确认后才显示终态。
- Turn 结束后尚未处理的审批变为 cancelled/expired。

## 5. 本地和远程竞争

用户可能同时打开本地页面和手机页面：

- 两端看到同一 `approval_id`。
- 第一个成功提交并被设备接受的决策获胜。
- 其他页面收到终态后立即禁用操作。
- 第二个决策返回“审批已经处理”，不能覆盖第一结果。

## 6. 离线和超时

- 设备离线后审批按钮禁用。
- Server 不把审批决定无限期排队。
- 决策命令有短过期时间，并绑定 `approval_id + turn_id`。
- 页面从后台恢复时先刷新审批状态再允许点击。
- 超时不明确时显示 unknown，不自动再次批准。

## 7. 通知与优先级

- 活跃页面顶部显示待审批数量。
- 当前 Thread 的审批在消息流中就地展示。
- 其他设备/Thread 的审批可在全局待办入口显示。
- 第一版不要求系统推送通知；后续可接 Web Push。
- 审批正文不出现在锁屏通知中。

## 8. 安全交互

- 批准和拒绝按钮视觉明确，避免误触。
- 高风险批准需要二次确认，具体由规范化风险等级决定。
- 页面必须显示目标设备，防止用户在错误电脑上批准。
- 截断内容必须标记，可请求更完整安全摘要。
- 不提供“全部自动批准”远程开关。

## 9. 验收标准

1. 审批能从 App Server 经 Agent、Server 到达本地和远程页面。
2. 两个页面并发决策只有一个成功。
3. 设备离线、Turn 结束或审批过期后不能继续批准。
4. 批准 HTTP 超时不会自动重复执行。
5. 审批终态以设备/App Server 确认为准。
6. 未识别审批默认不能批准，提示回本机处理或升级。
7. 审批日志不记录敏感完整正文。
