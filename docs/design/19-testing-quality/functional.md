# 测试与质量保障：功能设计

> 实现标记：本文描述目标设计；当前 `0.1.0` 已实现项和后续边界以 [后端实现状态](../../implementation-status.md) 为准。

## 1. 目标

从用户功能和稳定性承诺出发定义验收范围，确保系统不仅在正常网络下工作，也能在断线、重复、重启和部分故障后恢复到可信状态。

## 2. 核心验收旅程

### 首次接入

- 初始化 Owner。
- 生成并消费配对码。
- Agent 首次连接并上线。
- 本地添加 Project。
- 手机选择设备、浏览受控目录并创建另一个 Project。
- 手机看到 Project。
- 创建 Thread 和首个 Turn。

### 日常对话

- 恢复 Thread。
- 发送消息。
- 查看流式 Agent/命令/工具事件。
- Steering。
- Interrupt。
- Turn 完成。

### 审批

- 本地和远程同时看到审批。
- 一端批准，另一端同步终态。
- 设备离线/Turn 结束后审批不可操作。

### 故障恢复

- 手机切网、锁屏和刷新。
- Server 重启。
- Agent 重启。
- App Server 重启。
- Server/Client SQLite 短时故障。
- 事件重复、乱序、缺口。
- 设备离线时从 Server 阅读已同步完整历史；上线后继续增量回填。
- 目录查询期间设备断线、引用过期和目录被移动。

### 传输档位

- HTTPS/SSE + WSS 推荐档位完成全部核心旅程。
- HTTP/SSE + WS 兼容档位完成相同功能，同时持续显示不安全传输。
- WSS 证书失败不会静默降级到 WS。

## 3. 用户可见质量标准

- 不丢已接受命令。
- 不重复执行同一幂等命令。
- 不把连接断开显示成 Turn 失败。
- 不静默忽略事件 gap。
- 不把 unknown 显示为明确失败或成功。
- 不删除用户文件。
- 不泄露敏感凭证和正文到日志。
- 不把未完成历史回填显示为“全部已同步”。
- 不允许手机提交任意路径绕过目录选择器。

## 4. 支持矩阵

第一版至少覆盖：

- Agent：macOS、Linux。
- Browser：近期稳定版 Safari、Chrome/Chromium。
- 移动：iOS Safari、Android Chrome。
- Server：主流 Linux x86_64/arm64 之一或二者。
- SQLite/SQLx：Cargo.lock 固定版本及项目声明的最低 SQLite 能力。
- Codex：明确列出的 App Server Schema 范围。

Windows Agent 进入后续矩阵，不在第一版验收中伪装支持。

## 5. 发布等级

- Dev：核心流程可手动演示。
- Alpha：自动化主流程通过，允许已知恢复缺陷。
- Beta：故障注入和 24 小时长稳通过。
- Stable：支持矩阵、升级回滚、备份恢复和安全门禁通过。

## 6. 缺陷优先级

- P0：命令重复危险执行、未授权控制、数据删除、凭证泄露。
- P1：已接受命令丢失、审批错误、无法恢复、数据库损坏。
- P2：状态显示错误、部分事件缺失但可快照恢复、明显性能问题。
- P3：非关键 UI 和诊断体验。

P0/P1 阻止发布。

## 7. 验收标准

1. 所有核心旅程有自动化 E2E。
2. 每个跨进程边界有断线和重启测试。
3. 关键状态机有性质测试。
4. 稳定发布通过长稳、故障注入和升级回滚。
5. 支持矩阵之外的平台明确标记不支持。
6. P0/P1 为零才允许稳定发布。
7. 两种传输档位功能契约一致，但安全能力和风险提示符合各自档位。
8. 多设备完整历史、远程建 Project 和设备离线阅读都有自动化验收。
