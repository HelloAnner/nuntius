# 会话历史汇总与同步：功能设计

> 实现标记：本文描述目标设计；当前 `0.1.0` 已实现项和后续边界以 [后端实现状态](../../implementation-status.md) 为准。

## 1. 目标

把所有设备上的 Project、Thread、Turn 和 Item 汇总到公网 Server，使用户在手机上获得一个跨设备的完整历史视图。设备离线时可以阅读已经同步的全部历史；继续对话、审批、中断或执行命令仍需要原设备在线。

## 2. 归属关系

历史库必须严格保持：

```text
User
└─ Device
   └─ Project
      └─ Thread
         └─ Turn
            └─ Item
```

任何历史查询都可以沿层级过滤，并且每条记录都能反向定位所属设备和项目。第一版不把两台设备上的同名仓库或 Thread 自动合并。

无法安全匹配 cwd 的既有 Thread 归入该设备系统“未归类”Project。它仍会同步无路径的完整历史，从而满足“全部会话可见”，但在用户本地归类前不能继续执行。

## 3. Server 保存的完整历史

### 必须完整保存

- 用户发送的文本消息。
- Agent 完成后的消息全文。
- Thread、Turn 和 Item 的顺序与状态。
- Turn 的开始、完成、失败、中断和 unknown 状态。
- 命令、工具调用、文件修改和审批的结构化记录。
- 每条记录对应的设备、项目、Thread、Turn 和时间。

### 按策略保存

- 命令输出：在配置上限内保存全文；超大内容压缩、外置或明确截断。
- 文件 diff：只在 App Server 能稳定提供且用户策略允许时保存。
- 工具参数和结果：经过敏感字段过滤后保存。

### 不保存

- 项目源文件正文。
- Codex 登录凭证。
- 每一个逐 token delta 的永久副本。
- Agent/Server 普通调试日志作为会话历史。

流式 delta 用于实时展示和短期重放；Item 完成后以聚合后的最终内容作为长期历史。

## 4. 历史同步类型

### 4.1 实时增量同步

新 Turn 执行时，Agent 实时上传生命周期和 delta。Item 完成后上传最终 revision 和完整正文，Server 将临时实时状态固化为长期历史。

### 4.2 既有历史回填

Agent 第一次接入或升级到支持历史汇总的版本后：

1. 发现本地已有 Thread。
2. 按最近活动优先排序。
3. 分页读取 Thread 历史。
4. 批量上传规范化记录。
5. 保存每条 Thread 的同步 cursor。
6. 断线后从 cursor 继续。

回填不能阻塞当前对话、审批和中断等实时操作。

### 4.3 主动刷新

用户可以在设备或 Thread 页面请求“重新同步历史”。系统重新核对 revision，不重复创建记录。

## 5. 同步完整性状态

Thread 历史同步状态显示：

```text
not_started   尚未开始同步
syncing       正在回填
live          实时事件正在同步，但历史回填未必完成
complete      已知历史全部同步
partial       部分 Item 无法读取、被策略截断或版本不支持
error         同步遇到可见错误
```

实时新鲜度另行显示 `live/stale`。页面可以同时呈现“历史 complete；设备 stale；最后同步于 …”，设备离线不应把已确认的 complete 改成 partial，也不把已保存历史标记为不可阅读。

页面显示 `last_synced_at`、同步进度和 partial 原因。

## 6. 手机历史体验

- “全部会话”按最近活动聚合所有设备。
- 支持按设备、项目、状态、时间和标题筛选。
- Thread 卡片始终展示设备和项目归属。
- 打开 Thread 直接读取 Server 历史，不等待设备。
- 设备在线时，在 Server 历史上叠加活动 Turn 的实时事件。
- 设备离线时隐藏或禁用发送、审批和中断，但历史保持可读。
- 未归类 Thread 显示明显标签、不展示 cwd，并提供“请到对应设备本地归类”的说明。
- 长历史使用 cursor 分页和按需加载 Item 详情。

## 7. 重复与冲突

- 同一 Thread/Turn/Item 的重复上传只更新同一记录。
- 较旧 revision 不能覆盖较新内容。
- 内容哈希相同但重复上传不产生新版本。
- 同 ID 内容哈希冲突时保留最新可信 revision，记录同步冲突并要求重新核对。
- Server 历史不会反向改写 App Server Thread。

## 8. 删除与保留

- 归档 Thread 不删除 Server 历史。
- 撤销 Device 默认保留已同步历史，并标记设备已撤销。
- 用户可以按 Thread、Project 或 Device 删除 Server 历史副本；这不默认删除设备本地会话。
- 删除操作必须二次确认并说明影响范围。
- Server 历史保留策略可配置，默认长期保留。

## 9. 隐私提示

公网 Server 现在保存完整对话，部署和备份必须按敏感数据处理。HTTP 模式下，历史正文可能被网络窃听或篡改；页面必须持续显示风险，不能把 HTTP 描述为安全远程历史访问。

## 10. 第一版不做

- 跨设备合并同一逻辑 Thread。
- 在 Server 上直接运行或继续 Codex。
- 保存整个项目文件树或源文件正文。
- 对历史内容建立复杂语义搜索/向量数据库。
- 多用户分享历史。

## 11. 验收标准

1. 手机可以按设备和项目查看所有已同步 Thread 的完整消息历史。
2. 设备离线时仍可阅读 Server 历史。
3. 继续对话只会路由到 Thread 原设备。
4. Agent/Server 重启后历史同步从 cursor 继续。
5. 重复批次和重复 Item 不产生重复历史。
6. 实时 Item 完成后 Server 保存最终完整正文，而不是依赖 delta 永久拼接。
7. `complete/partial` 与 `live/stale` 两个维度真实可见，不互相覆盖。
8. 历史回填不会阻塞审批、Interrupt 和实时消息。
9. 未归类 Thread 也出现在全部历史中，但不能泄露 cwd 或绕过 Project 执行约束。
