# 模型套餐额度：功能设计

## 目标

在远程控制台的“设置”中集中查看每台已配对设备所登录的 OpenAI 与 Kimi Code 账户额度。额度属于设备上的本地账户上下文：两台设备可以使用不同账户，也可以使用同一账户，Server 不尝试合并。

## 用户能力

- Client 每个 UTC 小时至少采集并汇报一次 OpenAI、Kimi Code 额度快照。
- 设置页按设备分组，展示每个 Provider 最近一次快照；Server 没有数据时展示空状态。
- OpenAI 展示 5 小时、7 天窗口、余额、重置卡数量和下一张重置卡到期时间。
- Kimi Code 展示 5 小时、7 天窗口及 API 能确认的账户、权益和凭据到期信息。
- 账户信息包括可取得的账户 ID、邮箱、套餐、权益范围、会员起止/检查时间及凭据到期时间。上游没有提供的字段保持为空，不根据凭据到期时间推断会员到期时间。
- 用户可在设置页点击“刷新全部设备”。每台未撤销设备都会收到一条持久化刷新命令；在线设备立即执行，离线设备重连后执行。

## 历史语义

每一次自动或手动汇报都是独立快照。Server 永久追加保存，不按小时覆盖，也不因设置页只展示最新数据而删除旧记录。重复传输同一个事件是幂等重放，不产生第二行；新的汇报始终产生新行。

## 异常展示

- 没有凭据：`unavailable`，显示“未登录”。
- 凭据失效或上游拒绝：显示“登录已失效”。
- 网络失败或超时：显示“网络异常”。
- 额度成功但重置卡接口失败：`partial`，保留已经获得的窗口数据。
- 错误快照同样入库并成为最新状态，以免页面继续展示已经过时的成功结果。

## 验收标准

1. 两台设备使用不同账户时，设置页分别显示各自的最新账户与额度。
2. 同一设备连续汇报两次后，数据库保留两条记录，查询接口只返回第二条。
3. 无任何汇报时 `GET /api/v1/provider-usage` 返回 `[]`。
4. 手动刷新对所有未撤销设备创建命令，且刷新产生的新快照继续追加。
5. 桌面浏览器双列显示 Provider 卡片；窄屏浏览器切换为单列且不横向溢出。
6. Access Token、API Key、Cookie 和上游原始响应不进入事件、日志或 Server 数据库。
