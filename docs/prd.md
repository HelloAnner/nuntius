# Nuntius 产品需求文档

> 本 PRD 是产品目标。当前 `0.1.0` 后端的已实现能力、明确后续项和 HTTP 安全边界见 [后端实现状态与设计边界](./implementation-status.md)。

## 1. 文档信息

- 产品名称：Nuntius（暂定）
- 文档状态：需求已初步确认
- 更新日期：2026-07-18
- 产品定位：基于 Codex App Server 的、本地执行、多设备接入、服务端历史汇总、移动端远程控制平台

## 2. 背景与目标

用户可能同时在多台工作电脑上使用 Codex，并在不同电脑上维护不同项目。用户希望通过手机或平板浏览器访问统一的公网控制页面，查看自己的设备、项目和 Codex 会话，并远程发起或继续对话。

系统不是一个“云端版 Codex”，而是一套面向个人的多设备 Codex 控制平面：

- 每台工作电脑继续保存自己的项目、文件和 Codex 会话。
- 每台工作电脑安装 Nuntius CLI，由 CLI 管理本地 Codex App Server。
- 公网服务器负责身份认证、设备连接、状态管理、消息转发和所有设备会话历史的汇总存储。
- 手机或平板通过公网 Web 页面选择设备、项目和会话，并控制目标电脑上的 Codex。
- 真正执行 Codex、读写文件和运行命令的始终是被选择的工作电脑。

## 3. 产品原则

1. **本地执行、双层存储**：项目文件和 Codex 继续执行状态以设备本地为准；公网服务器保存规范化的完整会话历史，作为跨设备查询和离线阅读的数据源。
2. **项目是一等概念**：操作路径是“设备 → 项目 → 会话”，而不只是管理会话。
3. **设备主动连接**：设备端 CLI 主动连接公网服务器，不要求设备拥有公网 IP 或开放入站端口。
4. **服务端全局汇总**：公网服务器不保存项目源码，但完整汇总用户消息、Agent 回复、Turn/Item 结构化历史和必要执行摘要，使手机在设备离线时仍能查看已同步历史。
5. **复用 App Server**：使用 Codex App Server 管理 Thread、Turn、Item、审批和流式事件，不重新实现 Codex 会话引擎。
6. **本地与远程均可使用**：本地页面不依赖公网服务器也能完成本机项目和会话管理。

## 4. 整体架构

```text
手机 / 平板浏览器
        │ HTTPS/HTTP 命令 + SSE 事件
        ▼
公网控制服务器
  ├─ 用户登录与设备管理
  ├─ 连接路由与消息转发
  ├─ 远程控制 Web 页面
  ├─ 设备/项目/会话全局索引
  └─ 完整的规范化会话历史
        ▲
        │ 设备主动建立 WSS/WS 长连接
        │
 ┌──────┴──────────────┐
 ▼                     ▼
电脑 A                 电脑 B
Nuntius CLI            Nuntius CLI
├─ 本地管理页面         ├─ 本地管理页面
├─ 本地 SQLite          ├─ 本地 SQLite
├─ 项目目录             ├─ 项目目录
└─ Codex App Server     └─ Codex App Server
```

手机浏览器不直接连接工作电脑或 Codex App Server。每台设备上的 CLI 主动与公网服务器建立长连接，公网服务器在控制端与目标设备之间进行消息路由，并汇总设备上传的规范化会话历史。推荐使用 HTTPS/WSS；无 TLS 条件时兼容 HTTP/WS，但仅适用于可信网络、VPN 或用户明确接受风险的部署。

## 5. 系统组成

### 5.1 设备端 CLI

设备端使用 Rust 实现，同时承担本地 Agent、App Server 适配器和本地服务管理器的职责。

主要能力：

- 初始化并登录公网服务器。
- 为当前电脑注册唯一设备身份。
- 完成设备配对、注销和凭证管理。
- 启动、停止并检测本地 Codex App Server。
- 根据 Server 地址主动建立并维持 WSS 或 WS 长连接。
- 接收远程命令并转换为 App Server JSON-RPC 请求。
- 将 Codex 增量输出、工具调用、文件修改和审批请求实时传回控制端。
- 管理本机项目列表及项目配置。
- 向手机提供受控的目录浏览能力，使用户可以在选定设备后远程选择目录并创建项目。
- 索引本机已有 Codex 会话，并维护项目与会话之间的映射。
- 将已有和新增会话的规范化完整历史可靠同步到公网服务器。
- 使用本地 SQLite 保存设备配置、项目定义和会话映射。
- 启动本地 Web 页面，例如通过 `nuntius open` 打开。
- 支持后台运行、开机启动、断线重连、日志和状态诊断。

CLI 默认在本地启动并托管一个 Codex App Server，通过 `stdio` JSONL/JSON-RPC 与其通信；CLI 与公网服务器之间使用 Nuntius 自己的 WSS/WS 协议。Codex App Server 的 WebSocket 传输目前仍属于实验性能力，不应直接暴露到公网。

参考：[Codex App Server 官方文档](https://learn.chatgpt.com/docs/app-server)

### 5.2 公网控制服务器

公网控制服务器使用 Rust 实现。

主要能力：

- 用户登录和会话认证。
- 设备注册、配对、注销和密钥管理。
- 展示设备在线、离线、最后在线时间和版本信息。
- 接收并维护设备主动建立的长连接。
- 在手机网页与目标设备之间路由实时消息。
- 管理设备、项目、Thread、Turn 和 Item 的全局索引。
- 完整保存已同步的用户消息、Agent 回复和结构化会话历史。
- 提供远程控制 Web 页面、HTTP/HTTPS API、浏览器 SSE 事件流和设备 WS/WSS 接入端点。
- 控制访问权限，避免未授权客户端控制设备。
- 处理连接恢复、事件序号、重复消息、超时和心跳。
- 保存必要的安全审计记录。

公网服务器原则上不负责：

- 直接访问工作电脑文件系统。
- 直接运行 Codex。
- 保存项目源代码。
- 直接读取或修改 Codex 的本地数据库。
- 保存项目源代码和任意文件正文。
- 将服务端历史副本直接当作 Codex 的可执行状态；继续对话仍需目标设备和本地 App Server 在线。

### 5.3 Web 前端

前端使用常规 Web 技术开发，并使用 Bun 进行依赖管理和构建。产品包含两个独立入口；最终源码分别收敛在 `client/frontend` 与 `server/frontend`，不把根目录共享前端包作为运行时架构前提。可复制少量协议模型，但不能建立跨项目源码耦合。

#### 本地管理页面

- 由设备端 CLI 在 localhost 上提供。
- 只展示和管理当前电脑。
- 查看 Codex 安装、登录、App Server 和公网连接状态。
- 新增、编辑和移除本机项目。
- 查看、新建、恢复和归档本机会话。
- 进行本地对话和操作审批。
- 公网服务器不可用时仍可使用。

#### 远程控制页面

- 部署在公网服务器。
- 针对手机和平板进行响应式设计。
- 查看所有已注册设备及在线状态。
- 按“设备 → 项目 → 会话”进入操作界面。
- 选定在线设备后浏览该设备允许暴露的目录结构，并远程创建项目。
- 在指定项目中新建 Codex 对话。
- 查看所有设备已同步的完整历史会话；设备离线时仍可阅读，继续对话需要设备在线。
- 查看 Codex 实时输出、命令执行、工具调用、文件修改和状态。
- 远程批准或拒绝 Codex 的操作请求。
- 在不同设备和项目之间快速切换。

前端可以构建为两种运行模式：

- `local`：连接当前设备上的 Nuntius CLI。
- `remote`：连接公网控制服务器。

## 6. 核心领域模型

| 概念 | 含义 | 数据归属 |
|---|---|---|
| 用户 | 公网控制端的登录身份 | 公网服务器 |
| 设备 | 一台安装了 Nuntius CLI 的工作电脑 | 服务器登记，设备保存凭证 |
| 项目 | 某台设备上的一个工作目录或代码仓库 | 设备本地为路径权威，服务器保存全局索引 |
| Thread / 会话 | 一段连续的 Codex 对话 | 设备保存可执行状态，服务器保存完整规范化历史副本 |
| Turn | 用户的一次输入和 Codex 随后的执行过程 | 设备执行，服务器汇总状态和历史 |
| Item | 消息、命令、文件修改、工具调用等事件 | 设备产生，服务器保存规范化历史和必要内容 |
| 审批 | Codex 请求执行敏感操作时的确认 | 设备产生，控制端处理 |
| 连接 | 设备 CLI 与公网服务器之间的在线通道 | 双方临时状态 |

建议的主要关联关系：

```text
User
└─ Device
   └─ Project
      └─ Thread
         └─ Turn
            └─ Item
```

一个会话需要通过以下组合唯一定位到实际运行环境：

```text
device_id + project_id + app_server_thread_id
```

## 7. 项目模型

项目不是普通标签，而是设备本地的工作空间。项目至少包含：

```text
Project
├─ 所属设备 device_id
├─ 本地根目录 cwd
├─ 显示名称
├─ Git 仓库信息（可选）
├─ 默认模型和权限配置（可选）
├─ 关联的 Codex Threads
└─ 最近活动时间
```

项目规则：

- 项目必须属于一台具体设备。
- 每台设备有一个系统级“未归类”项目，不对应可执行 cwd，只用于承载尚未能安全映射到工作目录的既有 Thread；因此 Server 历史仍保持完整的 Device → Project → Thread 层级。
- 同一个 Git 仓库在电脑 A 和电脑 B 上，初期视为两个不同的项目实例。
- 新建会话时，由项目决定默认工作目录 `cwd`。
- “未归类”项目只能阅读和整理历史，Thread 关联到真实项目之前不能启动新 Turn。
- CLI 通过 App Server 获取会话，不直接依赖 Codex 内部 SQLite 数据结构。
- CLI 在自己的本地数据库中保存项目与 App Server Thread 的映射。
- 公网服务器保存 `device_id → project_id → thread_id → turn_id → item_id` 的归属关系和历史数据。
- 后续可以引入“逻辑项目”概念，将多台设备上的同一仓库关联起来，但不进入第一版。

## 8. 核心使用流程

### 8.1 设备接入

1. 用户在服务器页面登录并生成配对码。
2. 用户在工作电脑上运行 CLI 配对命令。
3. CLI 向服务器注册设备并在本地安全保存设备凭证。
4. CLI 启动 App Server 和本地服务。
5. CLI 主动连接公网服务器。
6. 手机页面显示设备在线。

### 8.2 本地使用

1. 用户运行 `nuntius open`。
2. CLI 启动或复用本地服务并打开浏览器。
3. 用户选择本机项目和会话。
4. 本地页面通过 CLI 调用 App Server。
5. App Server 的增量事件在本地页面实时展示。

### 8.3 手机远程创建项目

1. 用户在手机上选择一台在线设备。
2. 页面请求该设备的可浏览目录根。
3. CLI 只返回允许目录的按需、分页目录结构，不返回文件正文。
4. 用户逐层进入并选择目标目录。
5. 页面提交设备签发的短期目录引用和项目名称。
6. CLI 再次校验目录引用、路径权限和项目重复性。
7. 项目在设备本地创建，并将项目索引同步到公网服务器。
8. 用户进入新项目并创建 Thread。

### 8.4 远程发起新对话

1. 用户在手机上选择设备。
2. 用户选择目标设备上的项目。
3. 用户点击新建会话并输入文字。
4. 公网服务器把消息路由到目标设备。
5. CLI 调用 App Server 的 `thread/start` 和 `turn/start`。
6. App Server 在项目目录中执行任务。
7. CLI 将增量消息和执行事件实时返回手机。
8. 可执行会话保存在目标设备，规范化完整历史同步到公网服务器。

### 8.5 恢复历史会话

1. 用户打开目标设备和项目。
2. 服务器直接展示跨设备汇总的完整历史记录和同步状态。
3. 用户选择一个历史会话。
4. CLI 通过 App Server 恢复对应 Thread。
5. 后续对话继续在原设备执行，并增量同步到服务器。

### 8.6 远程审批

1. Codex App Server 产生审批请求。
2. CLI 将请求转发到远程控制页面。
3. 页面展示待执行命令、影响范围和可选决策。
4. 用户批准或拒绝。
5. 决策通过原连接返回 App Server。
6. App Server 继续或终止对应操作。

### 8.7 设备断线与恢复

1. 服务器通过心跳判断设备离线。
2. 远程页面仍可读取服务器已汇总的完整历史，并标记最后同步位置；不能继续执行新操作。
3. 离线期间不默认排队新的执行命令。
4. CLI 恢复网络后自动重连。
5. 双方根据连接会话、事件序号和快照重新同步状态。

## 9. 数据存储边界

### 9.1 设备本地保存

- 项目路径和项目配置。
- 完整 Codex 会话。
- 对话正文和执行事件。
- 文件修改结果。
- 本地 Codex 身份、配置和认证状态。
- 项目与 Thread 的关联关系。
- CLI 配置、设备凭证和本地运行状态。
- 尚未被公网服务器确认的历史同步 outbox。

### 9.2 公网服务器保存

- 用户账户。
- 设备身份、名称、状态和公钥或凭证摘要。
- 设备、项目、Thread、Turn 和 Item 的完整归属关系。
- 已同步的用户消息和 Agent 回复全文。
- 命令、工具、文件修改和审批的结构化历史及受控摘要。
- 每台设备和每条 Thread 的历史同步 cursor、完整性和最后同步时间。
- 连接状态和最后在线时间。
- 必要的安全审计记录。

### 9.3 离线数据表现

- 设备离线时，远程页面仍可显示服务器已经同步的完整会话历史。
- 页面必须标记历史是否完整、最后同步时间以及设备离线后的新增内容不可用。
- 离线设备不能继续真正对话或处理审批。
- 第一版不默认保存或排队离线指令，避免设备恢复后意外执行过期操作。

## 10. 功能需求

### 10.1 用户与认证

- 支持单用户账户登录。
- 支持安全退出和登录态失效。
- 支持查看和撤销已配对设备。
- 服务端不能将网页登录凭证直接作为设备凭证使用。

### 10.2 设备管理

- 配对新设备。
- 修改设备显示名称。
- 查看设备操作系统、CLI 版本和 Codex 状态。
- 查看设备在线状态和最后在线时间。
- 注销或撤销设备访问权限。
- 检测 CLI 与服务端协议版本兼容性。

### 10.3 项目管理

- 在设备本地添加项目目录。
- 在手机上选择目标设备，按需浏览允许暴露的目录并远程创建项目。
- 远程目录浏览只返回目录元数据和 Git 标记，不返回文件正文。
- 每台设备保留一个无 cwd 的系统“未归类”项目，承载无法安全映射目录的既有 Thread；历史可远程阅读，归类前不能继续执行。
- 编辑项目名称和默认配置。
- 移除项目索引，但不删除实际项目文件。
- 展示 Git 分支、仓库和脏状态等可选信息。
- 按最近使用时间排序项目。
- 展示项目关联的会话。

### 10.4 会话管理

- 在项目中创建新会话。
- 列出项目相关会话。
- 恢复历史会话。
- 展示会话标题、状态和最近活动时间。
- 归档和取消归档会话。
- 后续可支持基于已有会话创建分支会话。
- 在公网服务器按设备、项目和时间查看所有已同步会话的完整记录。
- 新设备首次接入或历史升级时，支持已有会话分页回填和断点续传。

### 10.5 对话与执行

- 支持发送文本消息。
- 支持流式展示 Agent 消息。
- 支持展示 Turn 状态。
- 支持展示命令执行、工具调用和文件修改事件。
- 支持中断当前 Turn。
- 支持在活动 Turn 期间追加指导信息。
- 保证同一会话的事件顺序可恢复、可去重。

### 10.6 审批

- 展示审批操作的类型、内容和影响范围。
- 支持批准、拒绝及 App Server 提供的其他决策。
- 处理审批超时、设备断线和页面关闭。
- 记录必要的审批审计信息，但避免记录敏感正文。

## 11. 非功能需求

### 11.1 安全

- 推荐公网通信使用 HTTPS/WSS。
- 兼容 HTTP/WS 部署；非 loopback 地址必须通过显式 `allow_insecure_http` 开启，并在页面和 CLI 持续标记“不安全传输”。
- HTTP/WS 模式只建议用于可信局域网、VPN 或 SSH 隧道；在不可信公网会暴露登录凭证、会话内容和控制命令。
- 每台设备使用独立、可撤销的设备凭证。
- 设备 CLI 只建立出站连接。
- 服务端必须验证用户是否有权控制目标设备。
- 敏感凭证不得通过 URL、日志或普通命令参数泄露。
- 服务端完整会话历史属于敏感数据，必须加密备份、最小权限访问并提供清理策略。
- 远程目录浏览限制在设备配置的允许根目录，禁止访问敏感系统目录和文件正文。
- 远程审批页面必须明确展示操作影响。
- 端到端加密可以作为后续增强能力。

### 11.2 可靠性

- CLI 支持自动重连和指数退避。
- 服务端支持心跳、连接超时和状态纠正。
- 消息需要唯一 ID、事件序号和幂等处理机制。
- 设备重连后能够恢复当前 Thread/Turn 的可见状态。
- 服务端重启不能导致设备身份丢失。

### 11.3 性能

- 对话消息和流式事件应保持低延迟。
- 单个设备的高频事件不能阻塞其他设备。
- 需要对事件流实施有界缓冲、背压和流量限制。
- 移动端应从公网服务器分页加载完整历史，避免一次加载所有会话和 Item。
- 历史同步按批次、cursor 和内容哈希执行，不能阻塞实时控制链路。

### 11.4 兼容性

- 设备端优先支持 macOS、Linux，后续支持 Windows。
- CLI 必须检测 Codex CLI/App Server 是否安装以及版本是否兼容。
- App Server 协议适配层应按 Codex 版本隔离。
- 可以通过 `codex app-server generate-json-schema` 生成与当前 Codex 版本匹配的协议 Schema，用于兼容性校验和类型生成。

### 11.5 可观测性

- CLI 提供运行状态、连接状态和最近错误信息。
- 服务端提供设备连接和消息路由指标。
- 日志默认隐藏提示词、凭证、文件内容等敏感信息。
- 支持生成便于排障的诊断报告。

## 12. 技术偏好

- 设备端 CLI：Rust。
- 公网控制服务器：Rust。
- 本地数据库：SQLite。
- 服务端数据库：第一版使用单活 SQLite，位于 Server 指定数据目录内，承载全局项目索引和完整会话历史；不依赖外部数据库。
- 前端：TypeScript 加常规前端框架，使用 Bun 构建。
- 实时通信：安全模式使用 HTTPS API + SSE、WSS；HTTP 兼容模式使用 HTTP API + SSE、WS。CLI 到 App Server 始终使用本地 stdio。
- CLI 到 Codex App Server：优先使用本地 `stdio` JSONL/JSON-RPC。
- 本地页面静态资源可以嵌入 CLI 二进制或随安装包分发。

## 13. 第一版 MVP

第一版包含：

- 单用户账户。
- 多设备配对、注销和在线状态。
- 每台设备管理多个项目。
- 项目绑定本地目录。
- 手机选择在线设备并浏览受控目录后创建项目。
- 新建、列出、恢复和归档 Codex 会话。
- 公网服务器汇总并完整保存所有设备已同步的会话历史。
- 已有本地会话的分页回填和断点续传。
- 文本对话与流式输出。
- 命令、工具调用和文件修改状态展示。
- 远程审批。
- 本地管理页面。
- 移动端远程控制页面。
- 断线重连和基本版本兼容检查。

第一版不包含：

- 项目文件跨电脑同步。
- 跨设备迁移 Codex 会话。
- 多人协作和复杂 RBAC。
- 手机读取或编辑任意本地文件；第一版只提供受控的目录选择器。
- 同一项目同时调度多台电脑执行。
- 默认离线任务队列。
- 跨设备逻辑项目合并。

## 14. 已确认的产品假设

1. 第一版以个人使用为主，支持单用户、多设备。
2. 项目属于具体电脑，本质上是一个本地目录或 Git 工作区。
3. 项目文件和 Codex 可执行状态保留在设备；服务器完整保存规范化会话历史，用于跨设备汇总和离线阅读。
4. 手机控制覆盖审批流程，而不只是收发聊天文字。
5. CLI 负责托管一个本地 App Server，并复用它承载多个项目和会话。
6. 设备离线时可以查看 Server 已同步的完整历史和持久索引，但实时设备状态为 stale，不能继续操作，也不默认排队任务。
7. 两套页面是两个独立产品入口，分别随 Client 和 Server 构建、部署；前端拆分由独立任务完成。
8. 第一版将公网服务器视为可信中继和敏感历史存储；端到端加密作为后续增强。
9. 支持 HTTP/WS 功能兼容，但明确不把它视为安全公网传输。

## 15. 成功标准

第一版完成后，用户应能：

1. 在至少两台电脑上安装并配对 Nuntius CLI。
2. 在手机浏览器中看到两台设备的实时在线状态。
3. 查看每台设备上独立的项目列表。
4. 在手机上浏览某台在线设备允许的目录并创建项目。
5. 在指定设备和项目中创建 Codex 会话。
6. 实时接收 Codex 的回复和执行事件。
7. 通过手机批准或拒绝需要确认的操作。
8. 设备离线时仍可在手机查看服务器已同步的完整历史会话。
9. 在公网服务器不可用时，仍通过本地页面使用本机 Codex。
